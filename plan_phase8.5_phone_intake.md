## 2. 배경 및 목표 (Goals)

### 2.1 배경

- 식당 사장님 고객층에 고령자가 많아 앱 설치·터치 입력보다 **전화 문의를 선호**한다. 현재는 대표가 전화를 직접 받아 수기로 접수하므로 (1) 통화 누락, (2) 접수 내용 유실, (3) 대표 1인 병목 문제가 있다.
- 기존 시스템은 앱 기반 ServiceRequest 접수만 지원한다. 전화로 들어온 문의가 시스템에 남지 않아 SLA 측정(금기 ④)·Hotspot 분석(blueprint §8.1-3)의 사각지대가 된다.

### 2.2 목표

1. 대표 문의번호로 온 전화를 **사람 개입 없이 자동 수신·녹음**한다.
2. 어르신은 평소처럼 말하게 두고 (완전 대화형 봇 금지 — 2.3 참조), 통화를 **녹음 + STT 텍스트화**만 한다.
3. 통화 내용으로 ServiceRequest **초안(DRAFT)**을 자동 생성한다.
4. 담당 지사에 FCM 푸시로 전달하되, **녹음 원본 + 변환 텍스트를 둘 다** 첨부한다.
5. 지사가 정보를 검토하고 식당에 **콜백**한다 (사용자가 이 운영 방식을 명시 승인함).
6. 전화 접수도 앱 접수와 동일하게 DB에 남아 SLA·통계·감사 대상이 된다.

### 2.3 설계 철학 — 왜 "대화형 봇"이 아닌가

- 고령자 음성은 사투리·잡음·맥락 생략이 많아, 봇이 되묻고 분기하면 어르신이 당황·이탈한다.
- 따라서 시스템은 **판단하지 않고 전달만 한다**: 녹음과 텍스트를 사람(지사)에게 넘기고, 최종 판단·콜백은 사람이 한다.
- 이는 blueprint §8.2 에이전트 가드레일("에이전트는 초안 생성까지만, 실제 액션은 사람 승인 후")과 동일한 원칙이다. STT·자동분류는 "초안"이고, 콜백·확정은 사람이다.

## 3. 동작 흐름 (Flow)

```
[1] 식당 사장님 → 대표 문의번호로 전화
        │
        ▼
[2] 클라우드 전화(가상번호) 자동 응답
    "안녕하세요 화이트온입니다. 문의하실 내용을 말씀해 주세요.
     말씀이 끝나면 끊으시면 됩니다." (안내멘트 1회, 되묻기 없음)
        │
        ▼
[3] 통화 녹음 (어르신 자유 발화, 봇 개입 없음)
        │
        ▼
[4] 통화 종료 → 클라우드 전화 업체가
    (a) 녹음 오디오 파일  (b) 발신번호  (c) 통화시각/길이  를 우리 백엔드 webhook으로 전달
        │
        ▼
[5] 백엔드 webhook 수신
    ├── 녹음 오디오 → S3 업로드 (DB엔 저장 안 함)
    ├── STT API 호출 → 텍스트 변환
    └── 발신번호로 Restaurant 조회 (phone 매칭)
        │
        ▼
[6] ServiceRequest 초안 생성
    - source = "PHONE", status = "DRAFT"
    - description = STT 변환 텍스트
    - 발신번호 매칭 실패 시 → 미등록 식당 처리 (8.X 참조)
    - PhoneCall 레코드 생성 (녹음 S3 URL + 텍스트 + 메타데이터)
        │
        ▼
[7] 담당 지사 결정 후 FCM 푸시 발송
    푸시 내용: "전화 문의 접수 — [식당명/발신번호]"
    지사 앱에서 녹음 재생 + 텍스트 열람 가능
        │
        ▼
[8] 지사가 녹음+텍스트 확인 → 식당에 직접 콜백
    콜백 후 지사가 ServiceRequest를 DRAFT → PENDING/ASSIGNED 로 확정
    (이 시점부터 기존 SLA·정산 흐름에 합류)
```

핵심: **[6] 단계까지는 전부 자동, [8]의 콜백·확정은 사람.** 시스템이 카테고리를 추정하더라도 그것은 초안일 뿐이고, 지사가 통화를 듣고 바로잡는다.

## 4. 기술 구성요소

| 구성요소 | 역할 | 우리 자산 여부 |
|---|---|---|
| 클라우드 전화 / 가상번호 ARS | 대표 문의번호 수신, 안내멘트 재생, 녹음, webhook 전송 | **신규 — 외부 계약 필요** |
| STT (음성→텍스트) | 한국어 녹음 → 텍스트 변환 | **신규 — 외부 API** |
| S3 객체 스토리지 | 녹음 오디오 원본 저장 | **보유** — `services/storage.py` `S3StorageService` 재사용 |
| 백엔드 webhook 엔드포인트 | 통화 종료 이벤트 수신, STT 호출, 접수 생성 | 신규 엔드포인트 (기존 FastAPI 라우터에 추가) |
| ServiceRequest 생성 로직 | 접수 초안 생성 | **보유** — `requests` 모듈 연동, `source` 필드만 추가 |
| FCM 푸시 | 지사 앱 알림 | **보유** — `notifications` 모듈 재사용 |

→ 실제 신규 개발은 **(1) webhook 1개, (2) PhoneCall 모델, (3) STT 클라이언트 래퍼** 3가지뿐이다. 나머지는 기존 자산 연동이다.

## 5. 국내 업체 옵션 비교

전화 수신과 STT는 (A) 한 업체가 둘 다 제공하거나, (B) 전화는 통신사/클라우드콜, STT는 별도 AI 업체로 조합하는 두 갈래다.

| 업체/방식 | 전화 수신 | 한국어 STT | 국내 적합성 | 비용 구조(대략) | 비고 |

### 5.2 권고

- **MVP: 네이버 클라우드(클라우드 전화 + CLOVA Speech) 단일 사업자.**
  - 이유 1: 전화 수신·녹음·STT를 한 콘솔/한 청구로 묶어 운영·정산이 단순하다 (소규모 사업장에 중요).
  - 이유 2: CLOVA Speech는 고령자·사투리 한국어에 대해 국내 STT 중 인식률이 가장 안정적이고, 화자분리·구두점·욕설/도메인 키워드 부스팅을 지원해 후속 자동분류 정확도를 끌어올린다.
  - 이유 3: 데이터가 국내 리전에 머물러 개인정보 국외이전 동의 이슈(legal-compliance)가 없다.
  - 이유 4: 우리 인프라(AWS)와는 별개지만, 녹음 webhook → 우리 S3 적재 구조라 클라우드 종속이 크지 않다.
- **Twilio 등 해외 업체는 채택하지 않는다** — 한국 전화번호 별정통신 규제, 개인정보 국외이전 동의 추가, USD 청구로 인한 회계 복잡성이 소규모 운영에 부담.
- 통신사 클라우드콜센터는 통화량이 월 수백 건을 넘어 "전담 상담 회선"이 필요해지는 시점(고도화 단계)에 재검토.

### 5.3 사람이 확인·결정해야 할 항목 (계약 전)

- 대표 문의번호: 기존 번호를 가상번호로 착신전환할지, 신규 1588/1666/050 번호를 개설할지.
- 녹음 파일을 webhook으로 **푸시**해 주는지, 아니면 우리가 **폴링/다운로드**해야 하는지 (업체별 상이 — webhook 푸시 가능 업체 우선).
- STT 결과를 업체가 함께 주는지, 우리가 오디오를 받아 별도 STT 호출하는지.
- 통화 녹음·STT 데이터의 업체 측 보관 기간 및 삭제 약관 (우리 보관 정책 §6과 정합 필요).
- 월 최소요금·약정기간·초기 설치비.

## 6. 녹음 저장 · 보관 설계

### 6.1 원칙 — DB 용량 효율화 (사용자 명시 요청)

기존 수리 사진 처리 방식(`RequestMedia` 모델 — DB엔 `file_url`만, 실제 파일은 S3)을 **그대로 답습한다.**

- 녹음 오디오 원본(.mp3/.wav 등) → **S3에만 저장.** DB에 바이너리(BLOB) 저장 절대 금지.
- DB에는 **S3 URL + 변환 텍스트 + 메타데이터(발신번호·통화시각·길이·STT 신뢰도)만** 저장.
- S3 경로 규칙: 기존 `S3StorageService`의 `folder` 인자를 활용해 `call-recordings/{uuid}.mp3` 형태로 분리 적재.

### 6.2 보관 기간 정책 (제안 — 사용자/legal-compliance 확정 필요)

| 데이터 | 보관 위치 | 보관 기간 | 만료 후 |
|---|---|---|---|
| 녹음 오디오 원본 | S3 | **2주(14일)** | 자동 삭제 (S3 Lifecycle Rule) |
| 변환 텍스트(STT) | DB (`PhoneCall.transcript`) | **영구** (ServiceRequest와 동일 수명) | 유지 |
| 통화 메타데이터(발신번호·시각·길이) | DB | **영구** | 유지 |

- 보관 기간 = **2주(14일)** — 대표 결정 (2026-05-16).
- 근거: 녹음 원본은 "지사가 콜백 전 통화를 직접 들어 정확히 파악"하는 용도이므로 접수 직후 며칠이 가장 중요하다. 2주면 콜백·초기 대응 확인에 충분하고, 그 이후엔 텍스트만으로 이력·통계·Hotspot 분석이 가능하다. 보관 기간이 짧을수록 개인정보 노출 위험도 줄어든다.
- 용량 효율: 2주 자동 삭제로 S3 비용이 무한 증가하지 않는다. 텍스트는 용량이 매우 작아 영구 보관해도 부담 없다.
- 구현: S3 버킷에 `call-recordings/` 프리픽스 대상 **Lifecycle Rule(14일 후 expire)** 설정. 별도 삭제 배치 코드 불필요 (devops-security 담당).
- 분쟁·법적 보관 필요 건은 예외적으로 별도 프리픽스(`call-recordings-hold/`)로 이동해 Lifecycle 미적용 — 정산 PDF의 5년 보관(blueprint §7.5)과 유사한 예외 처리.

### 6.3 개인정보·고지 (legal-compliance 협업 필수)

- 통화 녹음은 개인정보·통신비밀이다. 안내멘트에 **"상담 품질 향상 및 서비스 접수를 위해 통화가 녹음됩니다"** 고지를 반드시 포함한다 (멘트 1회, 발화 흐름은 방해하지 않음).
- 녹음·STT 처리 위탁(클라우드 전화·STT 업체)을 개인정보처리방침의 "처리위탁" 항목에 명시.
- S3 버킷은 비공개(Private). 녹음 재생은 지사 앱에서 **서명된 URL(presigned URL, 단기 만료)**로만 접근 — 공개 URL 금지.

## 7. 우리 DB 모델 영향

### 7.1 ServiceRequest — `source` 필드 추가

전화 접수와 앱 접수를 구분하기 위해 컬럼 1개를 추가한다.

```
ServiceRequest.source : str = "APP"   # APP | PHONE
```

- 기본값 `"APP"` → 기존 데이터·앱 접수 흐름에 영향 없음 (하위호환).
- `status`는 기존 값을 그대로 쓰되, 전화 접수는 `"DRAFT"`로 시작해 지사 콜백 후 `"PENDING"`/`"ASSIGNED"`로 승격. `"DRAFT"`가 기존 status enum에 없다면 추가.

### 7.2 신규 모델 — `PhoneCall` (RequestMedia 확장이 아닌 별도 모델 권장)

**판단: RequestMedia를 확장하지 않고 신규 `PhoneCall` 모델을 만든다.**

- 이유 1: `RequestMedia`는 수리 전/후 사진·동영상 증빙용이다. 통화 녹음은 성격(발신번호·STT 텍스트·통화길이·신뢰도 등 고유 메타데이터)이 달라, 한 테이블에 섞으면 컬럼이 대부분 NULL인 지저분한 스키마가 된다.
- 이유 2: 발신번호 매칭 실패(미등록 식당) 시 `request_id`가 아직 없는 통화도 저장해야 한다. `RequestMedia`는 `request_id` NOT NULL 전제라 부적합.

제안 스키마 (DDL은 backend-db가 Alembic 마이그레이션으로 확정):

```
PhoneCall
  id                UUID   PK
  request_id        UUID   FK service_requests.id  (nullable — 매칭 실패 시 NULL)
  caller_number     str    index            # 발신번호
  recording_url     str                     # S3 URL (오디오는 S3, DB엔 URL만)
  transcript        Text   nullable         # STT 변환 텍스트
  stt_confidence    float  nullable         # STT 신뢰도 (낮으면 지사에 '인식 불확실' 배지)
  call_started_at   datetime
  call_duration_sec int
  provider          str                     # 'naver_cloud' 등
  status            str    = "RECEIVED"      # RECEIVED | TRANSCRIBED | LINKED | UNMATCHED
  recording_expires_at datetime nullable     # 2주(14일) 후 — 만료 후 recording_url 무효 표시용
  created_at        datetime
```

- 모든 `PhoneCall` 생성·상태변경은 기존 `AuditLog`에 기록 (blueprint §7.5 / 금기 외 일반 규칙).
- Money 필드 없음 → Decimal 이슈 무관.

### 7.3 미등록 발신번호 처리

- 발신번호가 어떤 Restaurant와도 매칭되지 않으면: `PhoneCall.status = "UNMATCHED"`로 저장하고 ServiceRequest는 만들지 않는다. 본사(HQ)에 "미등록 번호 전화 문의" 알림 → 본사가 수동으로 식당 등록 후 연결하거나 직접 콜백.
- harnes.md §6(엄격 가입 검증)을 우회하지 않는다 — 전화 한 통으로 식당이 자동 가입되지 않는다. 등록은 여전히 정식 절차를 거친다.

## 8. 단계별 작업 순서 (Steps)

### 8.1 MVP (1차) — "녹음 받아서 사람에게 넘기기"

> 목표: 자동 분류 없이, 전화 → 녹음 → STT → 지사 전달까지의 파이프라인을 끝까지 한 번 관통.

1. **[사람]** 클라우드 전화 업체 선정·계약, 대표 문의번호 개설, 안내멘트 녹음(녹음 고지 포함). 비용 승인.
2. **[backend-db]** `ServiceRequest.source` 컬럼 추가 + `PhoneCall` 모델 신규 → Alembic 마이그레이션 1건.
3. **[devops-security]** S3 버킷에 `call-recordings/` 프리픽스용 Lifecycle Rule(14일 expire) 설정. STT/전화 업체 API 키를 `.env`로 주입(코드에 노출 금지). webhook 엔드포인트는 업체 IP/서명 검증.
4. **[backend-db]** 통화종료 webhook 엔드포인트 신규 개발: 녹음 → S3 적재(`S3StorageService` 재사용), STT 호출, 발신번호 매칭, `PhoneCall` + `ServiceRequest(DRAFT, source=PHONE)` 생성.
5. **[backend-db]** 담당 지사 결정 후 기존 FCM 푸시(`notifications`) 발송. 푸시 페이로드에 PhoneCall id 포함.
6. **[frontend-mobile]** 지사 앱에 "전화 문의" 접수 카드 추가: STT 텍스트 표시 + 녹음 재생(presigned URL). DRAFT → PENDING/ASSIGNED 확정 버튼. (기존 화면 보존 — `harness_rules.md`, 새 카드 추가 방식)
7. **[qa-analyst]** 실제 고령자 음성·잡음·사투리 샘플로 STT 인식률 검증. 미등록 번호·STT 실패·통화 0초 등 엣지케이스 테스트.

### 8.2 고도화 (2차) — 선택, 운영 안정화 후

8. **[Haiku 워커]** STT 텍스트 기반 카테고리 자동 추정(WATER_LEAK 등) → ServiceRequest 초안에 제안값 채움. **여전히 초안** — 지사가 수정. (blueprint §8.1-2 A/S 분류 에이전트와 통합 가능)
9. STT 신뢰도가 낮은 통화는 지사 앱에 '인식 불확실' 배지 표시 — 사람이 더 주의해서 듣게.
10. 전화 접수 SLA 별도 측정: webhook 수신 → 지사 콜백 완료까지의 시간을 통계화 (금기 ④ 정신의 전화 버전).
11. 반복 전화 문의 Hotspot 분석에 `source=PHONE` 데이터 합류 (blueprint §8.1-3).

## 9. 완료 조건 (Exit Criteria)

### MVP 완료 조건

- [ ] 대표 문의번호로 전화 시 안내멘트(녹음 고지 포함) 후 자동 녹음된다.
- [ ] 통화 종료 후 녹음이 S3에 적재되고, DB(`PhoneCall`)에는 URL·텍스트·메타데이터만 저장된다(바이너리 없음).
- [ ] STT 변환 텍스트가 `PhoneCall.transcript`에 저장된다.
- [ ] 발신번호가 매칭되면 `ServiceRequest(source=PHONE, status=DRAFT)` 초안이 생성된다.
- [ ] 담당 지사에 FCM 푸시가 가고, 지사 앱에서 녹음 재생 + 텍스트 열람이 된다.
- [ ] 지사가 DRAFT를 PENDING/ASSIGNED로 확정하면 기존 SLA·정산 흐름에 합류한다.
- [ ] 미등록 발신번호는 `UNMATCHED`로 저장되고 본사에 알림이 간다 (자동 가입 없음).
- [ ] S3 Lifecycle Rule로 14일 후 녹음이 자동 삭제되고, 텍스트·메타데이터는 유지된다.
- [ ] 개인정보처리방침에 녹음·STT 위탁이 명시된다.

### 고도화 완료 조건

- [ ] STT 텍스트 기반 카테고리 자동 추정 초안이 제공되되, 지사가 수정 가능하다.
- [ ] 전화 접수 SLA(webhook→콜백완료)가 측정·집계된다.

## 11. 우선순위 권고

**권고: 본 기능(Phase 8.5)은 결제·정산 안정화(Phase B / blueprint §7) 이후에 착수한다. 지금 당장 착수하지 않는다.**

근거:

1. **금기 ① 위반 회피**: blueprint의 핵심 순서는 "현장의 식당 앱·지사 앱(결제·출동 데이터)이 먼저 동작" 후 부가 기능이다. 전화 접수는 접수 채널 확장이지 결제·정산 코어가 아니다.
2. **인프라 선행 필요**: 본 기능은 S3 운영 전환(`STORAGE_TYPE=s3`)과 t3.small 업그레이드(blueprint §6.1)를 전제로 한다. 둘 다 결제 모듈 도입 시점에 함께 처리될 예정이므로, 그 흐름에 얹는 것이 효율적이다.
3. **사람 트랙 병목 없음**: 다만 클라우드 전화 업체 계약·번호 개설은 포트원/팝빌과 독립적이라, **사람 트랙(§5.3 항목)은 지금부터 병행 조사·견적 요청을 시작**해도 무방하다. AI 코드 착수만 Phase B 이후로 미룬다.
4. **ROI**: 고령 고객 비중이 높다면 전화 접수는 체감 가치가 크다. 단, 결제·정산이 불안정한 상태에서 접수 채널만 늘리면 처리 병목이 뒤로 밀릴 뿐이다. 코어가 안정된 뒤 도입해야 효과가 산다.

요약: **"업체 견적·번호 개설 조사는 지금 시작, AI 구현 착수는 결제·정산 안정화 후"** — 사람 트랙과 개발 트랙을 분리해 시간을 벌되 순서를 지킨다.
