## 1. 목표

식당·지사 가입 시 중복 판정 기준을 **사업자등록번호 + 휴대폰번호** 조합으로 바꾼다.
정상적인 가게(부부 공동운영·흔한 상호·한 건물 여러 점포)가 막히지 않도록.

## 2. 현재 문제 (코드 기준 — `restaurants.py` create)

가입 시 `이름`·`전화번호`·`주소`를 **각각 따로** 검사 → 셋 중 하나라도 겹치면 차단. 가장 빡빡한 방식.

| 검사 | 문제 |
| :--- | :--- |
| 이름 단독 차단 | "김밥천국" 등 흔한 상호는 전국 1곳만 가입 가능 (가장 심각) |
| 번호 단독 차단 | 부부가 한 번호 공유 시 둘째 가입 불가, 식당·지사 겸업 불가 |
| 주소 단독 차단 | 상세주소(호수)는 이미 포함돼 큰 문제 아님 |

## 3. 새 규칙 (확정안 — 사장님 초안, 아버님 승인 대기)

- **중복 판정 = `사업자등록번호 + 휴대폰번호` 조합.** 둘 다 같을 때만 "이미 가입된 가게"로 차단.
- **이름 단독 차단 제거**, **주소 단독 차단 제거.**
- 적용 대상: **식당·지사 양쪽 모두** (둘 다 사업체 → 각자 사업자번호 보유).
- 중복 검사 범위: 식당끼리 / 지사끼리. 같은 사업자번호로 식당·지사를 동시에 하는 것은 막지 않음.
- 사업자등록번호는 **가입 필수 입력** (10자리 숫자).

## 4. 기존 데이터

- **2026-05-17 운영·로컬 DB 전체 초기화 완료** → 기존 레코드 없음.
- 따라서 소급 입력(grandfathering) 불필요 — 새 규칙을 처음부터 전면 적용 가능.

## 5. 작업 항목 (승인 후 착수)

1. 모델: `Restaurant`·`Branch`에 `business_number: str` 컬럼 추가.
   - `init_db()`에 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS business_number VARCHAR;` (또는 Alembic).
2. 스키마: `RestaurantCreate`·`BranchCreate`에 `business_number` 추가 (필수).
3. 가입 검증 로직 교체 (`restaurants.py`, 지사 가입 경로):
   - 이름·주소 단독 차단 제거.
   - `business_number + phone` 조합 중복 검사로 교체.
4. 프론트: `restaurant.html`·`manager.html` 가입 폼에 사업자등록번호 입력란 추가 (10자리 숫자 형식 안내).
5. `harnes.md §6` 갱신 — 새 규칙 반영.

## 6. 완료 조건 (AC)

- [ ] 같은 주소·같은 상호라도 사업자번호가 다르면 정상 가입 (부부 공동운영 케이스)
- [ ] 같은 `사업자번호 + 휴대폰` 조합 재가입 시 차단
- [ ] 흔한 상호("김밥천국" 등)가 이름만으로 차단되지 않음
- [ ] 식당·지사 양쪽 가입 폼에 사업자번호 입력란 존재, 미입력 시 가입 거부
