# [실행 계획서] 전국 세척기 통합 관리 및 서비스 플랫폼 구축

본 문서는 프로젝트의 전반적인 방향성과 상세 스펙을 정의하는 마스터 청사진입니다. 주기적으로 업데이트되며, 모든 개발 및 운영 의사결정의 기준이 됩니다.

---

## 1. 프로젝트 개요 및 목적
본 프로젝트는 전국 단위의 세척기 판매, 수리(A/S), 세제 공급 및 부품 관리 사업을 디지털로 전환하는 것을 목적으로 한다. 식당(소비자)의 니즈를 실시간으로 수집하고, 전국 지사망을 통해 즉각적인 현장 서비스를 제공하며, 본사(마스터)가 전체 운영 현황과 실적을 데이터 기반으로 관리할 수 있는 통합 FSM(Field Service Management) 플랫폼을 구축한다.

## 2. 사용자별 핵심 기능 및 인터페이스 구성

### 2.1 소비자 (식당용 모바일 앱/웹)
- **원터치 서비스 요청**: 세척기 고장 수리, 세제 보충, 부품 교체 등 요청 사항을 사진/동영상과 함께 등록.
- **실시간 진행 현황 확인**: 접수 완료, 지사 출동 중, 수리 완료 등 서비스 전 과정을 앱에서 실시간 모니터링.
- **통합 결제 시스템**: 서비스 완료 후 수리비, 부품비, 세제 대금을 앱 내에서 직접 결제 (신용카드, 간편결제 등).
- **로열티 프로그램**: 결제 금액 기반 포인트 적립 및 재방문/재구매 유도를 위한 쿠폰 및 바우처 자동 발행.

### 2.2 지역 지사 (현장 대응용 모바일 앱)
- **실시간 긴급 알림**: 관할 지역 식당의 요청 발생 시 즉각적인 푸시(Push) 알림 수신.
- **출동 및 이력 관리**: 요청 수락 후 현장 방문, 조치 내용(사용 부품, 수리 내역) 등록 및 관리.
- **지사별 실적 대쉬보드**: 본인 지사의 매출 실적, 처리 건수, 정산 예정 금액 확인.

### 2.3 본사/마스터 (관리자용 PC 웹)
- **실시간 통합 관제**: 전국 지사의 대응 현황을 지도 및 리스트 형태로 모니터링.
- **SLA(Service Level Agreement) 관리**: 지사의 대응이 설정된 시간보다 늦어질 경우 경고 알림 표시.
- **데이터 분석 및 통계**:
  - 주별/월별/년별 지사별 매출 및 문제 유형 통계.
  - 동일 업체/지역의 반복적 고장 이슈(Hotspot) 분석 및 보고.
- **정산 관리**: 매주 단위로 발생하는 결제 대금에 대해 본사 수수료를 제외한 금액을 지사에 정산하는 기능.

## 3. 기술 스택 및 인프라 설계

### 3.1 소프트웨어 (SW)
- **Mobile App**: Flutter 기반 하이브리드 개발 (iOS/Android 동시 대응 및 실시간 푸시 기능 최적화).
- **Admin Web**: React.js 기반의 고성능 데이터 시각화 대쉬보드.
- **Backend**: FastAPI (Python) 기반의 고성능 API 서버.
- **Database**: PostgreSQL (고객 정보, 결제 이력, A/S 히스토리 관리).
- **Real-time**: Firebase Cloud Messaging (FCM)을 활용한 즉각적인 알람 시스템.

### 3.2 하드웨어 (HW) 및 클라우드
- **운영 서버**: AWS(Amazon Web Services) 기반의 클라우드 인프라 (안정성 및 확장성 확보).
- **데이터 파이프라인**: 대규모 데이터 연산 및 통계 분석, 향후 예측 분석을 위해 보유 중인 고성능 리눅스 서버(AMD Ryzen 9/Threadripper급)를 로컬 분석 노드로 활용하는 하이브리드 구성.

## 4. 비즈니스 모델 및 결제/정산 로직

### 4.1 결제 시스템 (PG 연동)
- **수수료 최적화**: 구글/애플의 인앱 결제 수수료를 피하기 위해 국내 PG사(포트원 등) API를 직접 연동.
- **결제 수단**: 신용카드, 계좌이체, 카카오페이, 네이버페이 등 지원.

### 4.2 수수료 및 정산 구조
- **정산 주기**: 주 단위 정산 (추후 운영 상황에 따라 월 단위 확장 가능하도록 유연하게 설계).
- **정산 흐름**:
  1. 식당 결제 시 대금은 본사 계좌로 취합.
  2. 시스템이 지사별/건별 매출을 자동 분류.
  3. 약정된 본사 수수료를 차감한 후 매주 지정된 요일에 각 지사 계좌로 자동 정산 리포트 생성 및 입금.

## 5. 단계별 실행 로드맵 (상세 실행 계획)

> [!IMPORTANT]
> **문서화 규칙**: 모든 세부 단계(Phase X.X) 착수 전에는 반드시 `plan_phaseX.X_description.md` 형태의 실행 계획서를 먼저 작성하여 히스토리와 설계 의도를 남긴다.

### Phase 1: 데이터 뼈대 및 백엔드 인프라 구축 (Backend & DB First)
* **1.1 DB 스키마 설계 (PostgreSQL):** 식당 정보, 지사 코드, 접수 이력, 부품 재고, 결제 내역을 담을 테이블 및 관계성(ERD) 설계.
* **1.2 코어 API 개발 (FastAPI):** 식당의 A/S 요청을 지사에 할당하고 결제 상태를 업데이트하는 핵심 비즈니스 로직 개발.
* **1.3 로컬 테스트베드 세팅:** 현재 구축되어 있는 고성능 로컬 서버(Ryzen 9 9950X / Threadripper 7970X 환경)를 활용하여 대용량 트래픽 및 데이터 파이프라인 처리 테스트 환경 구성.
* **1.4 외부 API 사전 연동:** 포트원(PortOne) PG사 가입 및 샌드박스 연동, Firebase(FCM) 프로젝트 생성.

### Phase 2: 현장 데이터 수집용 모바일 앱 MVP 개발 (App MVP)
* **2.1 식당(소비자)용 앱 (Flutter):**
    * 카테고리 선택형 A/S 접수 폼(사진 업로드 포함) UI 구현.
    * PG사 API를 통한 결제(신용카드/간편결제) 모듈 연동.
* **2.2 지사용 앱 (Flutter):**
    * FCM 기반 실시간 긴급 출동 푸시(Push) 알림 수신 로직 구현.
    * 접수 건에 대한 '출동 수락' 및 조치 후 '수리 완료(부품/세제 차감)' 상태 변경 기능.

### Phase 3: 데이터 기반 본사 대쉬보드 구축 (Web Dashboard)
* **3.1 실시간 관제 UI (React.js):** Phase 2에서 앱을 통해 쌓이기 시작한 '진짜 데이터'를 바탕으로 전국 지도 및 리스트 형태의 모니터링 화면 개발.
* **3.2 정산 및 통계 로직 연동:**
    * 매주 결제 대금에서 본사 수수료를 공제한 지사별 정산 리포트 자동 생성.
    * SLA(대응 지연 시간) 경고 알림판 및 동일 식당 반복 고장(Hotspot) 히트맵 구현.

### Phase 4: 파일럿 테스트 및 상용(Production) 배포
* **4.1 O2O 파일럿 테스트:** 본사 직영 지사 또는 통제가 수월한 지사 1곳과 연계된 식당들을 대상으로 실제 결제 및 출동 테스트 진행.
* **4.2 AWS 운영 환경 이관:** 로컬 서버에서 검증을 마친 백엔드 및 DB 시스템을 안정적인 AWS 클라우드 환경으로 마이그레이션.
* **4.3 전국 지사 롤아웃:** 앱 스토어(Google Play, App Store) 공식 심사 및 등록, 지사별 계정 발급 및 교육.
