# [실행 계획서] 전국 세척기 통합 관리 및 서비스 플랫폼 구축

본 문서는 프로젝트의 전반적인 방향성과 상세 스펙을 정의하는 마스터 청사진입니다. 주기적으로 업데이트되며, 모든 개발 및 운영 의사결정의 기준이 됩니다.

---

## 1. 프로젝트 개요 및 목적
본 프로젝트는 전국 단위의 세척기 판매, 수리(A/S), 세제 공급 및 부품 관리 사업을 디지털로 전환하는 것을 목적으로 한다. 식당(소비자)의 니즈를 실시간으로 수집하고, 전국 지사망을 통해 즉각적인 현장 서비스를 제공하며, 본사(마스터)가 전체 운영 현황과 실적을 데이터 기반으로 관리할 수 있는 통합 FSM(Field Service Management) 플랫폼을 구축한다.

## 2. 사용자별 핵심 기능 및 인터페이스 구성

### 2.1 소비자 (식당용 모바일 앱/웹)
- **원터치 서비스 요청**: 세척기 고장 수리, 세제 보충, 부품 교체 등 요청 사항을 사진/동영상과 함께 등록.
- **실시간 진행 현황 확인**: 접수 완료, 지사 출동 중, 수리 완료 등 서비스 전 과정을 앱에서 실시간 모니터링.
- **통합 결제 시스템**: 서비스 완료 후 수리비, 부품비, 세제 대금을 앱 내에서 직접 결제 (신용카드, 간편결제 등).
- **로열티 프로그램**: 결제 금액 기반 포인트 적립 및 재방문/재구매 유도를 위한 쿠폰 및 바우처 자동 발행.

### 2.2 지역 지사 (현장 대응용 모바일 앱)
- **실시간 긴급 알림**: 관할 지역 식당의 요청 발생 시 즉각적인 푸시(Push) 알림 수신.
- **출동 및 이력 관리**: 요청 수락 후 현장 방문, 조치 내용(사용 부품, 수리 내역) 등록 및 관리.
- **지사별 실적 대쉬보드**: 본인 지사의 매출 실적, 처리 건수, 정산 예정 금액 확인.

### 2.3 본사/마스터 (관리자용 PC 웹)
- **실시간 통합 관제**: 전국 지사의 대응 현황을 지도 및 리스트 형태로 모니터링.
- **SLA(Service Level Agreement) 관리**: 지사의 대응이 설정된 시간보다 늦어질 경우 경고 알림 표시.
- **데이터 분석 및 통계**:
  - 주별/월별/년별 지사별 매출 및 문제 유형 통계.
  - 동일 업체/지역의 반복적 고장 이슈(Hotspot) 분석 및 보고.
- **정산 관리**: 매주 단위로 발생하는 결제 대금에 대해 본사 수수료를 제외한 금액을 지사에 정산하는 기능.

## 3. 기술 스택 및 인프라 설계

### 3.1 소프트웨어 (SW)
- **Mobile App**: Flutter 기반 하이브리드 개발 (iOS/Android 동시 대응 및 실시간 푸시 기능 최적화).
- **Admin Web**: React.js 기반의 고성능 데이터 시각화 대쉬보드.
- **Backend**: FastAPI (Python) 기반의 고성능 API 서버.
- **Database**: PostgreSQL (고객 정보, 결제 이력, A/S 히스토리 관리).
- **Real-time**: Firebase Cloud Messaging (FCM)을 활용한 즉각적인 알람 시스템.

### 3.2 Infrastructure (AWS Cloud - ap-northeast-2)
- **Compute**: Amazon EC2 (Ubuntu 22.04 LTS) + Docker 기반 컨테이너화 운영.
- **Database**: Amazon RDS (PostgreSQL 15+) - Private Subnet 배치로 외부 접근을 차단하고 EC2 보안 그룹(SG)과만 연동.
- **Storage**: Amazon S3 (수리 전/후 사진 및 멀티미디어 파일 저장).
- **Networking & DNS**: 
  - **Route 53**: `whiteon` 자체 도메인 관리 및 DNS 라우팅.
  - **ACM (AWS Certificate Manager)**: SSL/TLS 인증서를 통한 HTTPS 암호화 통신 강제 적용.
  - **VPC**: 퍼블릭/프라이빗 서브넷 분리를 통한 네트워크 보안 계층화.
- **Hybrid Node**: 대규모 데이터 연산 및 통계 분석을 위해 기존 고성능 로컬 서버(Ryzen 9/Threadripper)를 로컬 분석 노드로 병행 활용.

### 3.3 Communication Logic (3-Way Ecosystem)
세 가지 주체 간의 실시간 데이터 흐름을 정의한다.
- **식당 사장님 (Owner)**:
  - 수리/세척/세제 요청 발생.
  - 실시간 기사 위치 및 도착 예정 시간(ETA) 조회.
  - 서비스 완료 후 결제 및 환불 요청.
- **지사/기사 (Branch/Driver)**:
  - 관할 지역 배정 알림(FCM Push) 수신 및 수락.
  - 수리 전/후 리포트 작성 및 사진 업로드.
  - 정산 내역 및 업무 스케줄 확인.
- **본사/HQ (Master Dashboard)**:
  - 전체 주문 흐름 및 지사별 가동률 관제.
  - 분쟁 발생 시 중재 및 정산금 확정/지급 승인.
- **통신 방식**: 
  - **REST API**: 기본적인 CRUD 및 비즈니스 로직 처리.
  - **WebSockets/FCM**: 배정 알림, 기사 위치 이동 등 실시간 상태 업데이트.

### 3.4 Security & Compliance
- **Security Groups**: 80/443 포트 외 외부 접근 전면 차단. DB 포트(5432)는 VPC 내부 EC2 소스에 대해서만 허용.
- **B2B Tax Automation**: 정산 단계에서 지사/식당 간 세무 처리를 자동화하기 위해 국세청 연동 전자세금계산서 API 연동 계획.
- **Data Privacy**: 개인정보보호법 준수를 위한 DB 암호화 및 접근 제어(IAM) 적용.

## 4. 비즈니스 모델 및 결제/정산 로직

### 4.1 결제 시스템 (PG 연동)
- **수수료 최적화**: 구글/애플의 인앱 결제 수수료를 피하기 위해 국내 PG사(포트원 등) API를 직접 연동.
- **결제 수단**: 신용카드, 계좌이체, 카카오페이, 네이버페이 등 지원.

### 4.2 수수료 및 정산 구조
- **정산 주기**: 주 단위 정산 (추후 운영 상황에 따라 월 단위 확장 가능하도록 유연하게 설계).
- **정산 흐름**:
  1. **식당 결제**: 서비스 완료 후 앱에서 결제 (본사 계좌 취합).
  2. **매출 분류**: 시스템이 지사별/건별 매출을 자동 분류.
  3. **본사 승인**: HQ 대시보드에서 정산 데이터 검토 및 지급 승인.
  4. **지급 및 증빙**: 본사 수수료 차감 후 지사 계좌 입금 및 자동 전자세금계산서 발행.

## 5. 단계별 실행 로드맵 (상세 실행 계획)

> [!IMPORTANT]
> **문서화 규칙**: 모든 세부 단계(Phase X.X) 착수 전에는 반드시 `plan_phaseX.X_description.md` 형태의 실행 계획서를 먼저 작성하여 히스토리와 설계 의도를 남긴다.

### Phase 1: 데이터 뼈대 및 백엔드 인프라 구축 (Backend & DB First)
* **1.1 DB 스키마 설계 (PostgreSQL):** 식당 정보, 지사 코드, 접수 이력, 부품 재고, 결제 내역을 담을 테이블 및 관계성(ERD) 설계.
* **1.2 코어 API 개발 (FastAPI):** 식당의 A/S 요청을 지사에 할당하고 결제 상태를 업데이트하는 핵심 비즈니스 로직 개발.
* **1.3 로컬 테스트베드 세팅:** 현재 구축되어 있는 고성능 로컬 서버(Ryzen 9 9950X / Threadripper 7970X 환경)를 활용하여 대용량 트래픽 및 데이터 파이프라인 처리 테스트 환경 구성.
* **1.4 외부 API 사전 연동:** 포트원(PortOne) PG사 가입 및 샌드박스 연동, Firebase(FCM) 프로젝트 생성.

### Phase 2: 현장 데이터 수집용 모바일 앱 MVP 개발 (App MVP)
* **2.1 식당(소비자)용 앱 (Flutter):**
    * 카테고리 선택형 A/S 접수 폼(사진 업로드 포함) UI 구현.
    * PG사 API를 통한 결제(신용카드/간편결제) 모듈 연동.
* **2.2 지사용 앱 (Flutter):**
    * FCM 기반 실시간 긴급 출동 푸시(Push) 알림 수신 로직 구현.
    * 접수 건에 대한 '출동 수락' 및 조치 후 '수리 완료(부품/세제 차감)' 상태 변경 기능.

### Phase 3: 기능 고도화 및 보안 강화 (Refinement & Security)
* **3.1 사용자 인증 및 권한 시스템 (Auth):**
    * JWT 기반 보안 로그인 구현 (식당용, 지사용, 본사용 권한 분리).
    * 지사별 관리자 계정 승인 및 관리 로직.
    * **[추가]** 서비스 이용약관 및 개인정보처리방침 초안 작성.
* **3.2 멀티미디어 데이터 처리:**
    * 수리 전/후 사진 및 동영상 업로드 기능 (`POST /media` 연동).
    * 이미지 리사이징 및 클라우드 스토리지(S3 등) 최적화 저장.
* **3.3 본사 관리 대시보드 (Web):**
    * 전국 지사 출동 현황 실시간 지도 관제 및 통계 그래프.
    * 주 단위 자동 정산 리포트 생성 기능.

### Phase 4: 상용 AWS 인프라 및 보안 구축 (Production Ready)
* **4.1 AWS 인프라 셋업:**
    * VPC(ap-northeast-2), EC2(Ubuntu 22.04), RDS(PostgreSQL 15) 인스턴스 생성 및 보안 그룹 설정.
    * Docker를 이용한 FastAPI 서버 컨테이너 배포 환경 구축.
* **4.2 도메인 및 SSL/TLS (HTTPS) 적용:**
    * Route 53을 통한 `whiteon` 도메인 연결 및 ACM SSL 인증서 발급/적용.
    * Nginx(또는 AWS ALB)를 통한 HTTP -> HTTPS 강제 리다이렉션 설정.
* **4.3 3-Way 실시간 통신 연동:**
    * FCM(Firebase Cloud Messaging) 및 WebSocket 서버를 활용한 앱간 실시간 상태 동기화 구현.
* **4.4 세무 자동화 API 사전 검토:**
    * B2B 정산을 위한 전자세금계산서 API 연동 규격 확인 및 테스트.

### Phase 5: 네이티브 앱 변환 및 최적화 (Mobile Engineering)
* **5.1 Capacitor 프로젝트 구성:**
    * 웹 코드(HTML/JS)를 안드로이드/iOS 네이티브 프로젝트로 변환.
    * 앱 아이콘, 스플래시 화면(로딩 화면) 디자인 및 적용.
* **5.2 네이티브 기능 연동:**
    * Firebase(FCM) 기반 실제 스마트폰 알림바(Push Notification) 연동.
    * 카메라 API 및 GPS 위치 정보 연동 최적화.
    * **[추가]** 네트워크 불안정 지역 대응을 위한 로컬 캐싱(Offline Support) 적용.
* **5.3 딥링크(Deep Link) 설정:**
    * 알림 클릭 시 특정 수리 요청 화면으로 즉시 이동하는 기능.

### Phase 6: 스토어 등록 및 최종 심사 (Release & QA)
* **6.1 구글 플레이 콘솔 준비:**
    * 구글 개발자 계정 생성 및 결제.
    * 개인정보처리방침(Privacy Policy) 최종본 등록 및 앱 내 링크 연결.
* **6.2 내부 테스트 및 QA (Closed Testing):**
    * **[중요]** 구글 플레이 신규 계정 정책에 따른 '20인 이상 테스터, 14일 유지' 요건 이행.
    * 실제 현장 버그 수정 및 앱 안정성(Crash rate) 관리.
* **6.3 공식 스토어 심사 제출:**
    * 앱 설명, 스크린샷 등록 및 최종 AAB(App Bundle) 업로드.
    * 구글 심사 대응 및 정식 출시 완료.
